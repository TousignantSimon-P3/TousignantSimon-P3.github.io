    Collaboration : Structure des Tests (3 Boîtes Isolées)

    Pour éviter que la compilation ne devienne complexe, on sépare le projet en trois entités indépendantes. Chaque dossier possède son propre bouton "Démarrer" (le main.cpp) et ses propres bibliothèques.

    1. Organisation des Fichiers (Soutien de l'Autonomie)

    Chaque dossier de test est une boîte fermée qui contient son propre point d'entrée et sa propre copie de Catch2.

    Projet/
├── .git/                    <-- RÉSERVÉ (Historique du projet principal)
├── .gitmodules              <-- Configuration des sous-modules Git (ex: test_commun)
├── CMakeLists.txt           <-- Orchestre les 3 cibles (app, test_perso, test_commun)
├── lib/                     <-- RÉSERVÉ (Vide pour le moment, pour futures libs globales)
├── include/                 <-- Pour les fichiers d'en-tête (.hpp)
├── src/                     <-- Votre code métier + main.cpp (APPLICATION)
└── tests/  
    ├── test_personnels/     <-- BOÎTE 1 : Vos tests personnels  
    │   ├── lib/  
    │   │   └── catch.hpp    <-- Catch2 pour vos tests  
    │   ├── main_test.cpp    <-- UNIQUE Moteur de test (le "main")
    │   ├── test_classe1.cpp <-- Tests pour la classe 1
    │   └── test_classe2.cpp <-- Tests pour la classe 2
    └── test_commun/         <-- BOÎTE 2 : LE SOUS-MODULE (Dépôt du groupe)
        ├── lib/ 
        │   └── catch.hpp    <-- Catch2 interne au groupe  
        ├── main_test.cpp    <-- Moteur de test du groupe  
        ├── test_classe1.cpp <-- Tests partagés pour la classe 1
        └── test_classe2.cpp <-- Tests partagés pour la classe 2

    1.1 Exemples de Contenu des Fichiers

    Voici exactement ce que doivent contenir vos fichiers pour que Catch2 fonctionne correctement sans erreurs de "double main" ou de compilation lente.

    A. main_test.cpp (Le Moteur UNIQUE)

    Ce fichier est compilé une seule fois. Il ne doit jamais contenir de tests (TEST_CASE). Son seul rôle est de générer le moteur de Catch2 pour TOUS les autres fichiers de tests. 

    #define CATCH_CONFIG_MAIN  // Génère le main() ICI SEULEMENT
#include "catch.hpp"

    B. Vos Fichiers de Tests (ex: test_classe1.cpp, test_classe2.cpp)

    Règle d'or : 1 classe = 1 fichier de test. Cela garde votre projet propre et facile à lire.

    Exemple : test_classe1.cpp 

    #include "catch.hpp"
#include "Classe1.hpp"

TEST_CASE("Test de base Classe1", "[classe1]") {
    Classe1 c1;
    REQUIRE(c1.fonctionAMethode() == true);
}

    Exemple : test_classe2.cpp 

    #include "catch.hpp"
#include "Classe2.hpp"

TEST_CASE("Test de base Classe2", "[classe2]") {
    Classe2 c2;
    REQUIRE(c2.getValeur() == 0);
}

    C. Les Fichiers de Tests Communs (Le Sous-Module)

    Le principe est exactement le même que pour vos tests personnels : 1 fichier par classe. Les fichiers sont simplement situés dans le dossier test_commun.

    Exemple : tests/test_commun/test_classe1.cpp 

    #include "catch.hpp"
#include "Classe1.hpp"

TEST_CASE("Test de base Classe1 (Commun)", "[groupe]") {
    Classe1 c1;
    REQUIRE(c1.fonctionAMethode() == true);
}

    Exemple : tests/test_commun/test_classe2.cpp 

    #include "catch.hpp"
#include "Classe2.hpp"

TEST_CASE("Test de base Classe2 (Commun)", "[groupe]") {
    Classe2 c2;
    REQUIRE(c2.getValeur() == 0);
}

    2. Configuration CMake (Symétrie Totale)

    Le script CMakeLists.txt à la racine orchestre les trois cibles de manière isolée.

    cmake_minimum_required(VERSION 3.10)
project(MonProjet)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# --- 1. LES SOURCES (La logique à tester) ---
file(GLOB TOUT_SRC "src/*.cpp")
set(SRC_FICHIERS ${TOUT_SRC})
list(REMOVE_ITEM SRC_FICHIERS "${CMAKE_CURRENT_SOURCE_DIR}/src/main.cpp")

# --- 2. VOS TESTS (Boîte Privée) ---
file(GLOB TEST_PERSO_FICHIERS "tests/test_personnel/*.cpp")

# --- 3. TESTS DU GROUPE (Boîte Commune) ---
file(GLOB TEST_COMMUN_FICHIERS "tests/test_commun/*.cpp")

# --- CRÉATION DES EXÉCUTABLES ---

# A. L'Application normale (./app)
add_executable(app src/main.cpp ${SRC_FICHIERS})
target_include_directories(app PRIVATE include)

# B. Vos tests personnels (./test_perso)
if(TEST_PERSO_FICHIERS)
    add_executable(test_perso ${TEST_PERSO_FICHIERS} ${SRC_FICHIERS})
    # On utilise le lib INTERNE au dossier personnel
    target_include_directories(test_perso PRIVATE include tests/test_personnel/lib)
else()
    message(WARNING "Dossier 'test_personnel' vide.")
endif()

# C. Les tests du groupe (./test_commun)
if(TEST_COMMUN_FICHIERS)
    add_executable(test_commun ${TEST_COMMUN_FICHIERS} ${SRC_FICHIERS})
    # On utilise le lib INTERNE au sous-module
    target_include_directories(test_commun PRIVATE include tests/test_commun/lib)
else()
    message(WARNING "Dossier 'test_commun' absent ou vide.")
endif()

    3. Pourquoi cette méthode ?

    1. Indépendance Totale : Le sous-module test_commun peut être cloné et testé seul sur un autre ordinateur. Il n'a besoin de rien d'extérieur à son propre dossier.
    2. Symétrie et Clarté : Vos tests personnels fonctionnent exactement comme ceux du groupe. Si vous savez gérer l'un, vous savez gérer l'autre.
    3. Le dossier lib/ à la racine : Pour le moment, ce dossier reste vide. Il servira plus tard dans la session si nous devons ajouter des bibliothèques globales qui servent à la fois à l'application ET aux tests (comme une bibliothèque JSON ou de gestion réseau).

    4. Rappel Git : Ajouter le Sous-Module

    git submodule add <URL_DU_DEPOT_DE_TESTS> tests/test_commun
git submodule update --init --recursive

    5. Workflow de Contribution (Scénario)

    ⚠️ Règle d'or : Le Contrat Social Puisque nous utilisons un dépôt privé sans restrictions techniques de GitHub, rien ne vous empêche techniquement de pousser directement sur main. Cependant, nous suivons un contrat social : tout changement doit passer par une branche et une Pull Request. C'est la seule façon d'apprendre le métier correctement et d'éviter de casser le code de vos collègues. Tout "push" direct sur main sera visible dans l'historique.

    Étape 1 : Se déplacer dans le sous-module

    Le dossier test_commun est un dépôt Git indépendant. Vous devez "entrer" dedans pour utiliser Git sur ce dépôt précis. 

    cd tests/test_commun

    Étape 2 : Créer une branche de contribution

    Ne travaillez jamais directement sur la branche principale (main). Créez une branche avec votre nom et le sujet du test. 

    git checkout -b ajout_test_validation_solde_simon

    Étape 3 : Ajouter votre test

    Copiez votre fichier .cpp (ou créez-en un nouveau) dans ce dossier. Rappel : Vérifiez que le test compile avec ./test_commun avant de partager !

    Étape 4 : Envoyer votre test au groupe

    git add mon_nouveau_test.cpp
git commit -m "Ajout du test de validation du solde"
git push origin ajout_test_validation_solde_simon

    Étape 5 : Créer la Pull Request (PR)

    Une fois votre branche envoyée sur GitHub : 1. Allez sur la page du dépôt de tests communs. 2. Un bandeau jaune devrait apparaître : "Compare & pull request". Cliquez dessus. 3. Vérifiez que la base est main et que votre branche est la source. 4. Écrivez un petit message expliquant ce que votre test vérifie. 5. Cliquez sur "Create pull request".

    Étape 6 : La revue et l'acceptation (Merge)

    C'est ici que le groupe doit s'organiser. Pour accepter une Pull Request : 1. Allez dans l'onglet "Pull requests" du dépôt. 2. Cliquez sur la PR à réviser. 3. Vérifiez les changements dans l'onglet "Files changed". 4. Si tout est correct, revenez à l'onglet "Conversation" et cliquez sur le gros bouton vert "Merge pull request". 5. Confirmez en cliquant sur "Confirm merge".

    Voici trois options de gestion pour votre groupe :

    • Option A : La Démocratie Ouverte Tout le monde a le droit de fusionner sa propre branche dans main. C'est rapide, mais attention : un seul test qui ne compile pas bloque tout le groupe !
    • Option B : Le Comité de Validation (Maintainers) Vous élisez 2 ou 3 "gardiens du code" qui sont les seuls autorisés à accepter les Pull Requests. Ils vérifient que le code est propre et qu'il compile avant de l'ajouter au main.
    • Option C : La Responsabilité Tournante Chaque semaine, un étudiant différent est responsable du dépôt de tests. C'est lui qui révise et fusionne les propositions de ses collègues.

    Étape 7 : Revenir à votre projet

    N'oubliez pas de ressortir du dossier pour revenir à la racine de votre projet principal. 

    cd ../..
