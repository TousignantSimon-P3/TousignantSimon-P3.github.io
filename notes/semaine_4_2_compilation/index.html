    L'Évolution d'un Projet C++ : Du Chaos à l'Automatisation

    Ce guide suit l'évolution naturelle d'un projet C++. L'objectif est de comprendre que chaque "bonne pratique" naît d'une douleur ou d'une répétition inutile.

    Étape 1 : Le "Vrac" (Tout à la racine)

    Au début, on met tout dans le même dossier. C'est simple, mais ça devient vite le bazar.

    Structure :

    ProjetBanque/  
├── main.cpp         <-- Point d'entrée de l'application (contient le main)  
├── Banque.cpp       <-- Implémentation de la classe Banque  
└── Banque.hpp       <-- Déclaration de la classe (interface)

    🔨 Compilation et Exécution :

    # Compiler  
g++ -Wall -std=c++20 main.cpp Banque.cpp -o app  

# Lancer l'application  
./app

    🔴 Le point de friction : Dès que vous atteignez 10 fichiers, chercher un header au milieu des fichiers compilés devient un enfer visuel.

    Étape 2 : L'Organisation (src et include)

    On sépare les intentions : ce qui est public (headers) et ce qui est privé (code).

    Structure :

    ProjetBanque/  
├── include/  
│   └── Banque.hpp   <-- On range les définitions (.hpp) ici  
└── src/  
    ├── main.cpp         <-- Point d'entrée principal  
    └── Banque.cpp       <-- Implémentation (.cpp)

    🔨 Compilation et Exécution :

    # Compiler (il faut dire au compilateur où est le dossier 'include')  
g++ -Wall -std=c++20 -I include src/main.cpp src/Banque.cpp -o app  

# Lancer  
./app

    🟠 Le point de friction : Le paramètre -I include devient obligatoire. On commence à "configurer" au lieu de juste "coder".

    Étape 3 : L'arrivée des Tests (Le point de rupture)

    On veut tester notre classe Banque de manière pro avec la bibliothèque Catch2.

    1. Télécharger la bibliothèque (Déjà existante)

    mkdir lib  
wget https://github.com/catchorg/Catch2/releases/download/v2.13.10/catch.hpp -O lib/catch.hpp <-- mettre le catch.hpp dans le dossier lib du projet

    2. Créer VOTRE point d'entrée (tests/test_main.cpp)

    #define CATCH_CONFIG_MAIN  
#include "../lib/catch.hpp"

    Structure complète :

    ProjetBanque/  
├── include/  
│   └── Banque.hpp       <-- Déclaration de Banque  
├── src/  
│   ├── main.cpp         <-- Possède le main() de l'APPLICATION  
│   └── Banque.cpp       <-- Code de Banque  
├── lib/  
│   └── catch.hpp        <-- La librairie téléchargée (ne pas toucher)  
└── tests/  
    ├── test_main.cpp    <-- Possède le main() des TESTS  
    └── test_Banque.cpp  <-- Vos TEST_CASE unitaires pour Banque

    🔨 Compilation et Exécution (Le cauchemar) :
    Il faut maintenant deux commandes différentes car on ne peut pas mélanger les deux fichiers contenant un main(). 

    # 1. Compiler l'Application  
g++ -Wall -std=c++20  -I include src/main.cpp src/Banque.cpp -o app  

# 2. Compiler les Tests (Notez l'ajout de -I lib)  
g++ -Wall -std=c++20  -I include -I lib tests/test_main.cpp tests/test_Banque.cpp src/Banque.cpp -o tests_app  

# Lancer les tests  
./tests_app

    🟡 Le point de friction : La commande est trop longue. Si vous ajoutez Compte.cpp, vous devez modifier vos deux commandes manuellement. C'est ici que l'on craque.

    Étape 4 : L'Automatisation Naïve (CMake rigide)

    Plutôt que de taper ces lignes de 100 caractères, on confie la "recette" à CMake.

    ⚠️ Nouveauté : Le dossier build/
    CMake génère énormément de fichiers temporaires (Makefiles, fichiers .o, caches). Si on lance CMake à la racine, on pollue tout notre projet.
    La règle d'or : On crée un dossier build qui servira de "poubelle de compilation". Tout ce qui est compilé ira dedans. Si on veut tout nettoyer, on supprime juste ce dossier.

    Fichier CMakeLists.txt :

    cmake_minimum_required(VERSION 3.10)  
project(ProjetBanque)  
include_directories(include lib)  

# On doit tout lister à la main... pénible si on a 50 fichiers !  
add_executable(app src/main.cpp src/Banque.cpp src/Compte.cpp)  
add_executable(tests_app tests/test_main.cpp tests/test_Banque.cpp src/Banque.cpp src/Compte.cpp)

    🔨 Compilation et Exécution :

    mkdir build      # 1. On crée le dossier de compilation  
cd build         # 2. On entre dedans  
cmake ..         # 3. On demande à CMake de préparer la recette (.. veut dire 'cherche le CMakeLists au dessus') il va créer le fichier makefile pour la compilation  
make             # 4. On lance la compilation réelle  

# Lancer l'un ou l'autre (ils sont maintenant dans build/)  
./app  
./tests_app

    Étape 5 : La Solution Flexible (Le Standard)

    On utilise le scan automatique (GLOB) pour ne plus jamais toucher au fichier CMake.

    Fichier CMakeLists.txt final :

    cmake_minimum_required(VERSION 3.10)  
project(ProjetBanque) 

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

include_directories(include lib)  

# On scanne les dossiers automatiquement  
file(GLOB SRC_FILES "src/*.cpp")  
file(GLOB TEST_FILES "tests/*.cpp")  

# On enlève le main de l'app pour les tests (pour éviter le conflit)  
set(LIB_FILES ${SRC_FILES})  
list(REMOVE_ITEM LIB_FILES "${CMAKE_CURRENT_SOURCE_DIR}/src/main.cpp")  

# On définit les deux exécutables  
add_executable(app src/main.cpp ${LIB_FILES})  
add_executable(tests_app ${TEST_FILES} ${LIB_FILES})

    Structure finale :

    ProjetBanque/  
├── build/               <-- CRÉÉ PAR VOUS (mkdir) : Contient tout ce qui est généré (app, tests_app, etc.)  
├── include/             <-- Tous vos headers (.hpp)  
├── src/                 <-- Toute votre logique (.cpp)  
├── lib/                 <-- Librairies externes (Catch2)  
├── tests/               <-- Tous vos fichiers de tests  
└── CMakeLists.txt       <-- Automatique et intelligent

    🔨 Utilisation au quotidien :

    cd build  
make             # Re-compile tout ce qui a changé  
./app            # Lance l'application bancaire  
./tests_app      # Lance la suite de tests

    🟢 Le résultat : Vous créez un fichier ? make. Vous ajoutez un test ? make. La structure est devenue invisible, robuste et automatique.
